You inspected the folder and decided these five files together were ready to record as a meaningful snapshot with a concise message like \"Initial site structure\".\n\nLater the team copied a final-export folder into the same place and, by accident, replaced index.html with a flattened file missing some local edits. Looking only at the final-export copy on disk, it's hard to see what was lost: the new filename and a duplicate folder name tell no story about what changed or why. A set of folders named final, final-v2, or final-july quickly becomes ambiguous.\n\nBecause you recorded the earlier snapshot, the project history stores a named moment that captured the exact contents of the five tracked files and the short message describing your intent. Looking at that snapshot (the commit) answers the concrete question: what did the project contain at the moment I said \"ready to record\"? The recorded snapshot contains file contents as a coherent package, not just a list of filenames; it preserves the exact text of index.html and other tracked files at that moment.\n\nRecovering from the overwrite is straightforward when you have a history: you inspect the history to find the commit message you wrote, open that snapshot to confirm it contains the expected content for index.html, and restore the tracked file from that snapshot. The repository history gives you a safe, explainable rewind point; duplicate final folders cannot provide the same context or selective restoration without guesswork.\n\nWhat one snapshot/commit row means: a single commit entry in the history represents a complete recorded state of the tracked files at one moment plus the message you wrote to explain intent. If you see a history row labeled \"Initial site structure,\" it means exactly the set of files you and your collaborators agreed were part of that meaningful milestone.\n\nA common mistake this example prevents: committing many unrelated edits at once (for example, content, style, and data changes in one snapshot) mixes motivations. The inspect → change → verify habit keeps commits focused so each history row remains useful as a narrative of the project."
